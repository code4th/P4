# 引き継ぎメモ: P1 / 自律成長AIエージェント基盤

Date: 2026-04-04

この文書は、Manager から与えられた P1 方針の固定版である。
原意を保持するため、内容は要約せず、解釈を混ぜずに保存する。

## 1. このプロジェクトの最上位目的

このプロジェクトの主目的は、AIフレームワークそのものを作ることではない。
主目的は、LLM が観測・知識化・批評・提案・評価・統治・自己改変を通じて、長期的に自律成長できる中核ループを作ること である。

そのうえで、将来的には以下を満たす人工生命的エージェント基盤へ発展させたい。

- ランタイム非依存
- モデル非依存
- OpenClaw 非依存
- ローカルLLM とクラウドLLM の役割分担
- 知識の蓄積より、知識の扱い方の成長を優先
- 長期 / 短期の統治構造を持つ
- 自己改変は監査・比較・巻き戻し可能である

## 2. 現時点の重要判断

採用済み方針

- OpenClaw は 暫定実行基盤 として使う
- ただし OpenClaw にロックインしない
- 中核ロジックは OpenClaw の外に置く
- OpenClaw は「手足」、外部コアが「脳と制度」
- ローカルLLMは補助脳として使う
- クラウドLLMは高品質判断系として使う
- 最初に作るべきは「知識を持つ頭」ではなく「知識の扱い方を自分で変えられる制度」

非採用 / 後回し方針

- 初手で独自フレームワーク全面実装
- 初手で OpenClaw 大改造
- 初手で本体LLMの頻繁な重み更新
- 初手で完全自律の自由改変
- OpenClaw のメインエージェントに内部から別エージェントを生成させる運用

## 3. なぜ OpenClaw を使うのか

ユーザーは複数モデルを柔軟に呼び出せる点で OpenClaw を現時点で有力視している。
また、API 課金の壁があり、サブスクで使える経路を活かしたい事情がある。

ただし、OpenClaw への依存は避けたい。
そのため、OpenClaw を母体にするのではなく、最初の実行ランタイムとして消耗品化する 方針である。

## 4. P1 とは何か

P1 は、既存メインBOTとは別個体の 独立成長AIエージェント の名前。
ユーザーは、既存メインBOT・P1・ユーザー自身の 3者構成を望んでいる。

P1 に求められているのは以下。

- 独立した会話インターフェースを持つ
- OpenClaw を通じて動くが、本体は外部コア
- ローカルLLM補助脳を使える
- 学び候補抽出、知識状態管理、批評ログ、運用ルール変更提案ができる
- 将来的に自律成長の中核個体になる

重要:
P1 は「OpenClaw の中の人格」ではなく、「OpenClaw を使う独立個体」として扱う。

## 5. OpenClaw 内部で直接エージェント生成しない理由

過去に、メインエージェントに別エージェントを作らせようとした際、セッション権限の問題で実行できないと言われた。
このため、以下の方針に変更している。

方針

- エージェントは、別エージェントの 仕様・設定・コード・手順 は作ってよい
- ただし、実体の生成・権限付与・起動は 外部 bootstrap が行う
- つまり「エージェントがエージェントを設計する」は採用
- 「エージェントがその場で内部権限だけでエージェントを生成する」は中核にしない

要するに、
P1 は「メインエージェントが産む」のではなく、「メインエージェントが設計し、外部スクリプトが産む」

## 6. 現在の最重要初動

最初にやるべきことは、OpenClaw に深く埋め込むことではなく、Python 製ローカルLLMワーカー を作ること。

採用済み構成

- OpenClaw: 入出力・ツール実行・OS操作
- Python ローカルLLMワーカー: Ollama 呼び出し
- 外部コア: 知識・運用・自己改善

まずワーカーに求める機能

- /summarize
- /classify
- /draft_lessons

役割

ローカルLLMに向く処理:

- 要約
- 分類
- タグ付け
- 学び候補の一次抽出
- ログ整形
- 前処理
- 埋め込み
- 夜間バッチ
- 反例候補列挙

クラウドLLMに残す処理:

- 最終的な制度変更
- 上位設計判断
- 高精度な長文抽象化
- 昇格 / 棄却の最終判断

## 7. 長期的に作りたい中核

最終的な成長コアの最小構成は以下。

- Event Log
- Knowledge Store
- Policy Engine
- Critic
- Proposer
- Evaluator
- Governor

知識状態は最低限以下で管理する。

- raw
- candidate
- deferred
- active
- retired

ここで重要なのは、
最初に自己改変させるのは知識ではなく運用ルール
であること。

## 8. 統治思想

このプロジェクトでは、知識の正しさ固定よりも 変化し続けられる制度 を重視する。
また、短期と長期のバランスを取るため、憲法・法律・運用に相当する多時間スケール統治を将来的に導入したい。

現時点で重要な原則は以下。

- 保留を消さない
- 反例を消さない
- ログを削らない
- 変更は比較評価する
- 巻き戻し可能にする
- 短期成功のみで長期ルールを壊さない

## 9. Codex にまずやってほしいこと

最優先

OpenClaw の詳細調査ではなく、外部コア化とローカルLLM補助脳の最初の実装骨格 を作ってほしい。

具体的には次の順番で進めてほしい。

Phase 1
Python 製ローカルLLMワーカーの最小実装

- ollama_worker.py
- HTTP API
- summarize
- classify
- draft_lessons
- JSON 入出力
- ログ保存
- エラー応答

Phase 2
P1 bootstrap の仕様

- P1 用 profile / workspace / prompt / config / runbook を外部で生成する枠組み
- OpenClaw 内部生成に依存しない構成

Phase 3
外部コアの最小ディレクトリ構成

- knowledge store
- policy engine
- critic
- proposer
- evaluator
- governor
- runtime adapter/openclaw

Phase 4
OpenClaw 側から Python ワーカーを使うための最小 adapter 戦略

- ただし OpenClaw 本体深改造はまだしない
- まずは外部サービス呼び出し前提

Codex には、理想論よりも 手順化可能な実装計画 を期待する。

欲しい成果物:

2. Python ワーカーの最初のファイル群

5. 「誰でも再現できる」 runbook
6. ロールバックしやすい進め方
7. OpenClaw 権限問題に引っかからない現実的な作業分担

## 11. 実装時の制約

必ず守ってほしいこと:

- OpenClaw 依存コードを中核に混ぜない
- OpenClaw を control plane として使う
- P1 本体を OpenClaw に埋め込まない
- ローカル / クラウド役割分担を崩さない
- 変更は最小単位で進める
- ログ、テスト、起動手順、ロールバック手順を毎回残す
- 将来 OpenClaw を捨てても中核が残る形にする

## 12. ユーザーの志向

- フレームワークより、自律成長が主眼
- ただし現実の経済制約も重視
- ローカルLLMを補助脳として導入したい
- 将来的なテクノロジー進歩を見越している
- OpenClaw は現実解として使うが、依存は避けたい
- 議論は広がりやすいので、Codex には整理しながら前進してほしい
- 将来的に「知能生態系」まで見ているが、今は最小成長ループから始めたい

補足:

- ユーザーのいう「コア完成」には、判断だけでなく、低リスクな範囲で自分で小さな改善実験を回し、そのフィードバックを次の更新に反映できることを含む
- つまり、完成後も外部実装者が毎回手で次フェーズを実装してやる前提にはしない
- 高リスク変更は承認対象のままでよいが、低リスク改善と小さな実験はコア自身が自律的に回せることを目標にする
