# P2 の問題はコンテキストの粗さか、tool loop 不在か

日付: 2026-04-11

## 結論

両方である。

ただし関係は上下ではなく連動している。

- コンテキストが粗い
- tool loop がない

この 2 つが組み合わさることで、同じ失敗を繰り返しやすくなる。

## 1. コンテキストが粗いとは何か

ここでいう粗さは「情報量が少ない」ではない。

本当に必要だったのは次の raw 因果列である。

- どのファイルのどの部分をどう書き換えたか
- その直後にどのコマンドで何が失敗したか
- その failure がどの変更と結びつくか

P2 は summary を多く持っていたが、この因果列が弱かった。

## 2. tool loop がないとは何か

tool loop とは、モデルが次のような小さな行為を明示的に選び、その結果を次の判断材料へ戻す形である。

- `read_file`
- `grep`
- `run_test`
- `edit_file`
- `finish`

重要なのは「ツールを使うこと」自体ではなく、

- 行為が小さい
- 結果が構造化される
- そのまま履歴に残る

ことである。

## 3. P2 は何が違ったか

P2 は phase ごとに大きな prompt を作り、

- 今の状況を要約
- 何を考えたかを要約
- 変更後の全文を書かせる

という呼び方をしていた。

そのため、モデルは 1 回の呼び出しの中で同時に

- 状況理解
- 原因推定
- 修正案作成
- コード全文生成

をまとめてやることになった。

これは「全部自分でやる」に近い。

## 4. 何がまずかったか

### A. 小さく確認せずに大きく書き換える

tool loop 型なら、

1. `read_file`
2. `run_test`
3. failure 確認
4. `edit_file`
5. `run_test`

と刻める。

P2 は `revised_file_content` に全文を書かせるため、間違うとファイル全体を壊す。

### B. 失敗が「説明」になり、行為ログにならない

tool loop 型では失敗そのものが observation になる。

P2 では failure は validation report として残るが、LLM の主たる入力は要約された bundle だったため、行為と failure の連続性が弱くなった。

### C. 自分のミスを局所化しにくい

tool loop 型では

- どの command で壊れたか
- どの patch の後に壊れたか

が狭く特定される。

P2 では 1 回の生成責務が重いため、ミスの責任範囲が広すぎた。

## 5. つまり何が問題だったか

「ツールを使わなかったこと」そのものが悪ではない。

本質的に悪かったのは、

- 小さな再現可能な行為に分けず
- その結果を raw のまま次へ戻さず
- 1 回の生成でまとめて解こうとした

ことである。

tool loop はその問題を避けやすい標準形に過ぎない。

## 6. P2 に必要だったこと

P2 に本当に必要だったのは、

- raw な edit/action/result の列
- それを継承する session
- 必要ならそこから子フレームへ降りる再帰

である。

つまり順番は、

1. session/action/result loop
2. その上に再帰階層コンテキスト

であり、逆ではなかった。
