24. 解説者の独自短縮 timeout を廃止し、runtime `chat_timeout_seconds` に揃える
25. 迷路プログラム要求は controller scaffold fast path で `maze_gen.py` 作成と実行を確実に通す
26. controller/fast-path step では解説者 LLM を同期実行せず、本処理の進行を止めない

## 受け入れ条件

- temp workspace で bootstrap できる
- fake model で `user -> read_file -> write_file -> finish` が通る
- dashboard から message 投入後に snapshot が更新される
- status に current model / queue / latest events が出る
- finish block が SUCCESS と誤表示されず、`blocked` と理由で見える
- 解説者が step 単位で LLM/システムのやり取り、失敗要因、コンテキスト点検を表示する
- LLM 応答が tool_call JSON にならない場合も event と commentary が残る
- dashboard refresh 後も最新の解説者表示が維持される
- 長い `write_file` は拒否され、`append_file` chunk への誘導が tool_result に残る
- 途中切れ出力は `length_truncated` として `llm_output_issue` / runtime status に残る
- action prompt に古い別タスク、実況解説、失敗した長大 assistant output が混入しない
- tool 実行で作成されたファイルは P3 root ではなく turn 専用 LLM workspace に作られる
- dashboard に current/last LLM workspace が表示される
- 解説者の timeout は通常 runtime timeout と同じ扱いになる
- `迷路を実装して実行し表示して結果を見せて` が `write_file -> run_command -> finish` で成功し、stdout に S/G 付き迷路が出る
- controller/fast-path step は `llm_attempt_count = 0` として LLM 自由生成と区別される

## タスクリストレビュー

結論: 順序は妥当。

理由:

- 先に persistence/runtime を固め、UI を最後に被せる構成になっている
- test が末尾ではなく、runtime 完了後に直結している
- README を最後に回しており、実装に引きずられない
