# AIに「なぜ動かないか」を聞き続けた3週間──P2の顛末

前回の記事では、自己改善 AI（P2）が「問題を分解し始めたが、戻ってこられない」という話を書きました。今回はその後の話です。

でも今回は設計思想の話ではなく、**私と AI コーディングエージェントの間で実際に何が起きていたか**を書きます。ログと会話記録を掘り返して、3週間の経緯を俯瞰します。

## やっても、やっても、うまくいかなかった

4月上旬。P2 は 220 回動いて 42 回昇格し、v0044 まで到達していました。前半は順調でした。`self_check()` の改善、診断の具体化、運用者メッセージの整備。自己改善の初歩としては悪くなかった。

でもそこからが問題でした。後半の 64 回は一度も昇格しない。同じ SyntaxError を繰り返す。diff が 0 行なのに「修正した」と言う。止まっているのか壊れているのか分からない。

それで私は AI に聞きました。「なぜ動かないのか、調べてくれ」と。

## 4月9日：最初の解析セッション

AI コーディングエージェント（Codex）に P2 のログを渡して、長時間 run の分析を依頼しました。

エージェントは丁寧に調べて、こういう報告を返しました。

> P2 は合計 220 回動き、42 回採用され、115 回失敗し、63 回は途中で終わったままになっていました。

> 後半に増えた失敗は、ほとんど変わっていない、変えたけれど壊れた、変えたけれど価値が小さすぎた、の 3 つでした。

> 朝 6:39 以降、monitor ログには次の記録が何度も出ていました。worker probe failed 3x; restarting loop worker。この再起動は 63 回ありました。途中で終わったままの試行も 63 回なので、長く考えている途中の処理を監視側が「止まった」と誤認して落としていた可能性が高いです。

つまりエージェントは「AI の問題だけじゃない、監視側も悪い」と指摘しました。私はこの分析をもとに修正し、P2 を再稼働させました。

でもまた止まりました。

## 4月11日深夜：一晩で12本の設計メモ

そこで私はもう一度 AI に聞きました。今度はもっと根本的に。「OSS の強い agent と比べて、P2 は何が違うのか」と。

AI エージェントはその夜、一晩で **12 本の設計メモ** を書き上げました。ファイルのタイムスタンプが全部を語っています。

```
00:19  note-self-improvement-ai-hierarchy（記事ドラフト）
00:44  claw-code-purpose-achievement-analysis（claw-code 分析）
00:55  open-source-agent-purpose-execution-comparison（OSS 横断比較）
00:58  p2-minimal-kernel-cut（最小 kernel 提案）
01:25  p2-meta-design-clarification（設計思想の整理）
01:28  p2-why-repeats-failures（失敗原因の正本）
01:33  p2-context-vs-tool-loop（コンテキスト問題の分析）
01:37  p2-context-failure-ack（実装側の失敗認識）
01:39  p2-what-to-do-next-meta（次にやるべきこと）
03:20  p2-recursive-frame-rules（再帰フレーム運用規則）
11:57  P2_INDEX + p2-context-carryover-design（インデックス＋文脈設計）
16:12  p2-session-kernel-implementation（新 kernel 実装記録）
```

深夜0時から朝の4時にかけて 9 本。翌日にさらに 3 本。OSS 6 製品の実装を比較し、失敗原因を診断し、設計案を書き、メモの重複や矛盾を整理するインデックスまで作った。

エージェントの結論はこうでした。

> P2 は「失敗を直す agent runtime」ではなく「失敗を説明する phase pipeline」に寄っていた。

> 問題は再帰階層コンテキストの発想が悪かったのではなく、それを支える内側 kernel が session/tool-result 型になっていなかったことにある。

つまり P2 が実験していた再帰階層コンテキスト──フレーム階層でコンテキストを分解・継承・回収する仕組み──はアイデアとしては正しいが、それを載せる session loop がなかった、と。

## エージェントが書いた「完璧な計画」

その夜のうちに、AI エージェントは新しい kernel の実装まで終わらせました。`session_action_loop_v1`。テストは 36/36 全通過。

```
- 子フレームが親の観測結果を継承できる ✓
- 子フレームが return_payload を親に返せる ✓
- 親フレームが子の返却後に続行できる ✓
- 各 LLM 呼び出しの再現材料が残る ✓
```

完璧に見えました。

でも実際には、この新 kernel は一度も live workspace に切り替えられませんでした。

## 切り替えられなかった理由

ここが今回いちばん大事なところです。

AI エージェントは 12 本のメモと新しい kernel を一晩で作りました。分析は鋭く、実装はテストも通っていました。**でも、それを既存のシステムに実際に組み込んで動かす最後の一歩が、踏めなかった。**

なぜか。

理由は単純です。P2 のコードベースは 12,700 行に膨らんでいて、`loop.py` だけで 2,821 行。その上に 20 以上のモジュールが依存していました。新 kernel に切り替えるには、この巨大な legacy との結合を解く必要がありました。

AI エージェントは、分析と新規実装はできます。でも「巨大な既存システムの中で、壊さずに切り替える」という実作業は、一晩の集中セッションでは終わらなかった。そしてその後、私は P2 を一旦リセットして、旧来の kernel のまま動かし直してしまいました。

## リセット後、昇格ゼロ

4月16日に P2 をリセットしました。bootstrap seed の状態からやり直し。

結果は、13 回の試行で昇格ゼロ。`Broken pipe` で落ちるもの、validation に失敗するもの、子フレームに降りたまま帰ってこないもの。

リセット前は 19 回昇格していました。リセット後はゼロ。蓄積された状態に依存した偶発的な成功だったのかもしれない。

## 「解析→修正→また止まる→また解析」の無限ループ

3週間を俯瞰すると、こういうパターンが見えます。

```
P2が止まる
→ AIエージェントに「なぜ？」と聞く
→ エージェントが的確に分析する
→ エージェントが修正案を出す（ときには実装までする）
→ 修正を入れる
→ P2がまた止まる
→ またAIエージェントに聞く
→ ……
```

AI エージェントの分析は毎回正しかったと思います。「監視の誤停止が原因だ」──正しい。「session loop がない」──正しい。「全文書き換えが壊れやすい」──正しい。「再帰は実行プリミティブにすべきだ」──正しい。

でも、**正しい分析を積み重ねても、動くものにならなかった**。

なぜか。

## AI エージェントが得意なことと苦手なこと

ここで見えてきたのは、AI コーディングエージェントが協力者として得意なことと苦手なことの境界です。

**得意だったこと：**
- ログを読んで何が起きたか分析する
- OSS の構造を比較して本質的な差を見抜く
- 設計メモを体系的に書く
- 新しいモジュールをゼロから実装する
- テストを書いて通す

**苦手だったこと：**
- 12,700 行の既存コードの中で、壊さずに切り替える
- 「完璧な計画」を「動くデプロイ」に変える最後の一歩
- 「分析は正しいが実行に至らない」ループを断ち切る判断

AI エージェントは新しいものを作るのは速い。でも既存の複雑なシステムとの integration は、依存関係の全体像を同時に握り続ける必要があり、1回の会話セッションでは完結しにくい。

もうひとつ気づいたのは、**AI エージェントは「やめる」とは言わない**ということです。「分析をやめよう」とか「P2 をここで止めよう」とは提案してこない。聞かれれば応える。計画を立てろと言われれば完璧な計画を出す。でも「もうこの方向はダメだ、別のことをしよう」とは、求められない限り言わない。

だから、やめる判断は人間がしなければならなかった。

## P2 を止めた

4月17日以降、P2 の goal status は `paused` になっています。

止めた理由は、P2 が悪かったからではなく、**解析と計画の無限ループを断ち切るため**でした。

AI エージェントが一晩で書いた 12 本のメモの中に、すべての答えはありました。「再帰は正しいが、session loop が先」「summary より event」「思考を教えるのではなく、気づける世界を与える」。

でもそれを既存の P2（12,700 行）に後から組み込むのは、分析を繰り返すだけでは到達できない。だから P2 の経験を踏まえた新しいカーネルを、P3 としてゼロから切り出すことにしました。

AI エージェントの分析は正しかった。実装能力も高かった。足りなかったのは、**「この方向のままでは動くものにならない」と人間が見切ること**でした。

## まとめ

P2 の3週間は、人間と AI コーディングエージェントが協力して自己改善 AI を作ろうとした実験でした。

AI エージェントはログを的確に分析し、OSS と比較し、設計メモを 12 本書き、新しい kernel まで実装しました。でも、動くものにはならなかった。

分析が正しくても、実行に至らないことがある。完璧な計画が、完璧なデプロイにならないことがある。AI エージェントは「やめよう」とは言わない。解析→修正→また止まるという無限ループを切れるのは、人間の「見切り」だけです。

P2 で学んだいちばん大事なことは、技術設計の話ではなく、**AI と一緒にものを作るときに、人間がどこで判断を下すべきか**ということでした。
