# 自己改善AIは「同じ失敗を繰り返している」と気づけるのか

前回の記事では、自己改善 AI を長時間動かすと、前半はかなり伸びるものの、後半で止まりやすいことを書きました。今回はその続きです。止まったあとに見えてきたのは、もっと根本的な問題でした。自己改善 AI にとって本当に重要なのは、「自分を書き換えられること」だけではなく、「自分が今どう詰まっているのかに気づけること」ではないか、という話です。

## いまの自己改善AIはどこまで進んでいるのか

今回見ている実験系は、執筆時点で `v0071`、generation `71`、cycle_count `70` でした。完全に止まっているわけではありません。実際、試行全体の内訳も、promoted `70`、rejected `67`、started のまま `12` で、失敗ばかりではありませんでした。

直近の履歴だけ見ても、そのことは分かります。

```json
{"generation": 69, "step": "promotion", "candidate_id": "c0143", "message": "candidate c0143 promoted to v0070"}
{"generation": 70, "step": "validation", "outcome": "failed", "candidate_id": "c0144"}
{"generation": 70, "step": "validation", "outcome": "failed", "candidate_id": "c0145"}
{"generation": 70, "step": "promotion", "candidate_id": "c0148", "message": "candidate c0148 promoted to v0071"}
```

つまり最近の自己改善 AI は、「同じところで完全停止」ではなく、失敗が続く区間がありつつ、ときどきそこを抜けて昇格する動きをしています。

## きっかけは、同じ SyntaxError の繰り返しだった

その中で目立っていたのが、同じような SyntaxError を何度も繰り返していたことでした。`c0120`, `c0121`, `c0122` はほぼ同じ箇所で落ちていました。

```text
File ".../agent/goal_logic.py", line 182
    parser.add_argument("--d..."
                       ^
SyntaxError: '(' was never closed
```

`c0127` では少し形が変わったものの、やはり同じ系統でした。

```text
File ".../agent/goal_logic.py", line 195
    parser.add_argument("--d...
                        ^
SyntaxError: unterminated string literal (detected at line 195)
```

ここで重要なのは、単に `validation failed` ではないことです。実際には、同じファイルの、かなり近い場所を、似た形で壊し続けていました。つまり「最近 SyntaxError が多い」ではなく、「`goal_logic.py` の `main()` 末尾付近を似た形で壊し続けている」と言えます。

## 面白かったのは、AI自身も少しずつ気づき始めていたこと

もっと面白かったのは、こちらがそれを見抜く前に、AI 自身もかなり近いことを言い始めていたことです。`c0127` の `situation_report` には、こういう記録が残っていました。

```json
{
  "known": [
    "過去6回の試行はすべてSyntaxErrorで失敗",
    "agent/goal_logic.pyのself_check関数がエラー発生時にログ出力を行う",
    "テストは現在のファイル内容で成功している"
  ],
  "suspected": [
    "構文エラーを引き起こす変更を繰り返している",
    "検証前に構文の妥当性を確認するプロセスが欠如している"
  ]
}
```

さらに `pre_edit_reflection` では、こう書いていました。

```json
{
  "what_kept_happening": "変更がSyntaxErrorで失敗し続け、探索が停滞している。",
  "what_this_suggests_about_my_search": "構文エラーを引き起こす変更を繰り返し試している。",
  "what_must_be_different_this_time": "コードの構文を確認する前段階を設ける必要がある。"
}
```

まだ完全ではありません。でも少なくとも、同じ失敗が続いていること、探索が停滞していること、今のやり方に問題があること、までは自分で言葉にし始めています。

## ただ、気づくだけではまだ足りなかった

とはいえ、その気づきがすぐに改善成功へつながるわけではありませんでした。たとえば `c0122` では、かなりまともな反省を書いています。

```json
{
  "what_kept_happening": "構文エラーが連続して発生し、探索が停滞している。",
  "what_i_might_be_missing": "コードの構文チェックや現在の状態把握が足りない可能性がある。",
  "what_must_be_different_this_time": "変更前に構文確認を行う必要がある。"
}
```

でも結果はこうでした。

```json
{
  "candidate_id": "c0122",
  "status": "rejected",
  "decision_reason": "validation failed"
}
```

つまり今の自己改善 AI は、「何が起きているか」はかなり分かる一方で、「その気づきを次の行動制約に変える」ことがまだ弱い段階にいます。

## それでも前進はしていた

ここで面白いのは、完全に停滞したわけではないことです。`c0143` では、AI はかなり具体的に問題を言い換えていました。

```json
{
  "known": [
    "ファイルにはSyntaxErrorが存在する",
    "main関数の定義が不完全である可能性がある"
  ],
  "chosen_response": "main関数の定義を修正し、文法エラーを解消する"
}
```

この試行は実際に昇格しました。

```json
{
  "candidate_id": "c0143",
  "status": "promoted",
  "decision_reason": "candidate validated and active goal retry passed"
}
```

つまり AI は、同じ失敗を何度も繰り返しつつも、ときどきそこを抜けて前進しています。一直線に成長するのではなく、失敗が続き、少し気づき、また失敗し、あるところで通る、というぎこちない学習のような挙動になっています。

## なぜ同じ失敗を繰り返しやすいのか

今回のケースで大きかったのは、前回の失敗の詳細が次の試行に十分引き継がれていなかったことです。一般的な AI エージェントは、直前の作業文脈や失敗内容がそのままコンテキストに残っていれば、同じ失敗を避けやすくなります。今回の自己改善 AI でも反省文は残っていましたが、次の一手を強く縛る形では渡せていませんでした。そのため、「どこを」「どう壊したか」という具体的な失敗の情報が薄まり、同じ種類の失敗を反復しやすくなっていたと考えています。

## 次にやること

次にやることははっきりしています。コンテキストを引き継ぐ対応を入れることです。ただし、単に前回の全文を長く渡せばいいわけではありません。長く渡しすぎると、かえって焦点がぼけます。

そこで考えているのが、再帰的にコンテキストを引き継ぐ形です。まず現在の目的を置く。大きすぎるなら一段細かい目的に分ける。その階層で必要な失敗情報だけを持つ。やるべきことをやる。まだ足りなければもう一段潜る。終わったら一段上に戻る、という流れです。

このとき各階層に渡すのは、全文の履歴ではなく、次の行動を変えるための差分です。たとえば今回の SyntaxError なら、「最近構文エラーが多い」では弱い。「`goal_logic.py` の `main()` 末尾付近を壊した」「閉じ括弧の欠落が続いた」「前回と同型なのでそこは触り方を変えるべき」といった形で引き継ぐ方が効きます。

## まとめ

今回分かってきたのは、自己改善 AI に必要なのは、単に自分を書き換える能力ではないということでした。本当に大事なのは、自分が今どう詰まっているかに気づくこと、その気づきを次の行動に変えること、そして必要なコンテキストを切らさずに短い修復ループを回せることです。

今の AI はまだ途中です。でも少なくとも、「最近失敗が多い」ではなく、「自分の探索様式が停滞している」と自分で言い始めています。自己改善の前に、自己理解がいる。今回の run は、そのことをかなりはっきり見せてくれました。
