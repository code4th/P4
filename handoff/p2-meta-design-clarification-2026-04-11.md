# P2 Meta Design Clarification

Date: 2026-04-11

Purpose:

- `P2` の本来の設計意図を、実装側の判断と混同しない形で固定する
- これまでの設計経緯と、今回 OSS 調査に至った理由を明文化する

---

## 1. 主語を分ける

ここで必ず分けるべき主語は 2 つある。

### 1.1 `P2` 自身

- 自己を改良する AI エージェント
- 高レベル目標は「自分自身をより高性能にすること」
- 外界への操作だけでなく、自己変更も目的達成手段として扱う

### 1.2 `P2` を実装する側

- `P2` の仕組みを設計・実装するエンジニアリング側
- ここでは Codex がその役を担う
- `P2` の思考と、`P2` を実装する側の思考は別である

この区別を崩すと、

- `P2` が考えるべきこと
- kernel が固定で持つべきこと
- 実装者が外から判断してしまっていること

が混ざり、設計が歪む。

---

## 2. `P2` の本来の中心命題

`P2` の中心命題は次である。

- `P2 は自分を改良する AI エージェントである`

副次的に coding agent 的な性格を持つのではなく、

- 自己改善が主
- コード編集はそのための主要手段の一つ

という順序で理解する。

つまり `P2` は、

- 「コードを書くエージェント」ではなく
- 「自己改善のためにコードも書くエージェント」

である。

---

## 3. 以前ぶつかった本質的問題

`P2` は自己改善を回そうとしていたが、実装エラー、とくに構文エラーや壊れた編集を自力で安定して直せなかった。

その本質原因として見えていたのは次である。

- `P2` が自分で何を変えたかの把握が粗い
- その直後に何が壊れたかの把握が粗い
- どの変更とどの失敗が結びつくかの因果が粗い

つまり問題は、単に「反省が浅い」ことではなく、

- 自分の行為の raw 情報
- その結果の raw 情報

が、次の思考へ十分な解像度で渡っていなかったことにある。

---

## 4. その解決として導入しようとしたもの

この問題に対する本質的な解決として、次の 2 つを入れようとしていた。

### 4.1 因果の分かるコンテキスト引き継ぎ

必要なのは要約を増やすことではなく、少なくとも次を `P2` 自身が参照できることだった。

- 何を書き換えたか
- その直後に何が壊れたか
- それがどの変更と結びつくか

要するに、

- action raw
- result raw

を因果が見える形で次回へ渡すことが必要だった。

### 4.2 再帰階層コンテキスト

サブエージェントや問題分解を、個別機能の寄せ集めではなく本質的に扱う手段として、再帰階層コンテキストを導入しようとしていた。

意図は次である。

- 問題が大きすぎれば局所 goal へ分解する
- 親文脈を保持したまま子階層へ降りる
- 局所修復や調査を終えたら親へ戻る
- これを再帰的に繰り返せるようにする

つまり、

- 問題分解
- サブエージェント的分業
- 文脈継承

を 1 つの再帰フレーム機構として統一的に扱うことが狙いだった。

---

## 5. それでもうまくいかなかった理由

再帰階層コンテキストやスキル誘導を入れても、期待したほど機能しなかった。

その理由は、少なくとも次の 2 つに整理できる。

### 5.1 runtime の主役がまだ `P2` 自身ではなかった

現在の `P2` は、

- kernel が phase を決める
- LLM は各 phase で JSON を返す

という構造が強く、`P2` 自身が

- 道具を選ぶ
- 結果を受ける
- 次の行動を決める

runtime の主役になり切っていなかった。

### 5.2 コンテキストが会話 loop ではなく phase ごとの再構成だった

必要だったのは、

- 自分の行為
- その結果
- 次の判断

が 1 本の session/history の中で持続することだった。

しかし実際には、

- context_selecting
- reflecting
- generating

のような phase ごとに再構成された prompt が中心になっていたため、因果が弱くなった。

---

## 6. だから今回 OSS 調査を行った

上記の問題を踏まえ、一般的な AI エージェントが「どうやって目標達成の loop を成立させているか」を確認するために OSS 調査を行った。

調べた意図は次である。

- 目標達成の本体が何かを確認する
- みな同じように prompt を工夫しているだけなのかを確認する
- 実際には runtime kernel に何を持っているのかを確認する

調査の結果、共通していた本質は、

- `messages/history`
- `tools`
- `tool results`
- `stop/finish`

を持つ loop であり、賢い説明文や誘導文そのものではなかった。

---

## 7. 現時点の設計上の重要結論

現時点で固定すべき結論は次である。

### 7.1 `P2` の主語

- `P2` は自己改善エージェントである

### 7.2 解くべき本質問題

- 自分の変更と失敗結果の因果を、自分で追えるようにすること

### 7.3 問題分解の本質手段

- 再帰階層コンテキストを使って、局所 goal へ降り、親文脈へ戻ること

### 7.4 kernel に必要なこと

- `P2` 自身が tools と results を扱える session/runtime loop を持つこと

### 7.5 今回の OSS 調査の意味

- 現在の `P2` は prompt/workflow 側に寄りすぎており、
- 本来必要な runtime loop が弱いことを確認するための調査だった

---

## 8. 今後の設計で守るべきこと

今後 `P2` を再設計する際は、少なくとも次を崩してはいけない。

- `P2` の高レベル目標は自己改善である
- `P2` の思考と、実装者の思考は分ける
- 因果の見える raw context を次回へ渡す
- 再帰階層コンテキストを、問題分解と文脈継承の基本機構として扱う
- その上で、`messages/tools/tool results` を持つ runtime kernel を作る
