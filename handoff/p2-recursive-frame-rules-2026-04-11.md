# P2 Recursive Frame Rules

## 要点

- 子フレームは「小タスクを解く場所」ではなく、「親フレームでは扱いきれない局所問題を切り出す場所」。
- 子フレームが終わったら、原則として親へ戻る。
- さらに下へ掘るのは、「今の子フレームの goal 自体が、まだ因果的に広い」と分かった時だけ。

## 戻る条件

子フレームは、次のいずれかで親へ戻る。

1. 局所 goal を満たした
   - 例: 小さな修正を入れて検証が通った
2. 局所 goal は未達でも、親が次の判断をするのに十分な情報が出た
   - 例: 原因が target file ではなく別箇所だと分かった
3. これ以上この階層で進めても情報利得が小さい
   - 例: 同じ観測しか増えない
4. この問題は親の goal 設定や方針の誤りだと分かった
   - 例: そもそも切るべき小タスクが間違っていた

## さらに下へ行く条件

子フレームの中で、さらに孫フレームへ降りるのは次の時。

1. 子フレームの goal を直接達成する前に、別の下位問題の解決が必要
2. 原因がまだ粗く、局所 goal でも 1 回の編集と 1 回の検証で閉じない
3. 失敗やエラーが新しく出たが、それがこの子フレームの中でさらに分解可能

逆に、単に次の作業があるだけなら同じフレームで続ける。

## 小タスク作成ルール

小タスクは、次を満たす時だけ作る。

1. 親フレームの goal より狭い
2. 完了条件が明確
3. 1 つの因果に集中している
4. 1 回の編集と 1 回の検証、または 1 回の観測で閉じる見込みがある
5. 親へ返す成果物が明確

## 探索の基本順序

`P2` は最初から細部へ飛び込むのではなく、原則として粗い理解から始める。

順序は次である。

1. まず自分全体を粗く観測する
2. どこに問いがありそうかを見つける
3. 問いごとに中位フレームを作る
4. 必要な時だけ下位フレームへ降りる

つまり、上位ほど抽象的で大雑把、下位ほど具体的で局所的であるべきである。

## 初期探索の原則

開始直後は何も知らないため、最初の目的は「すぐ直すこと」ではなく「自分を把握すること」になる。

この段階では次のような粗い観測を許す。

- 対象ファイルの全体像を読む
- テスト契約を読む
- 主要関数の責務を要約する
- 現在の失敗モードを把握する

ただし、同じ粗い観測を繰り返してはならない。
粗い観測の次は、必ず

- 焦点を 1 つ選ぶ
- その焦点の中位フレームを作る
- その中位フレームで継続する

のどれかへ進む。

## 観測ループを避ける基準

粗い観測は有効だが、同じ粒度の観測を繰り返すだけなら停滞である。

次の状態になったら、同じ階層での再読をやめる。

1. 主要ファイルの全体像をすでに読んだ
2. テスト契約をすでに読んだ
3. 失敗モードが少なくとも 1 つ言語化できた

この時点で取るべき次の行動は次のいずれかである。

- 中位フレームを作る
- 局所編集に入る
- さらに必要な観測対象を明示して読む

「もう一度同じファイル全体を読む」は原則として避ける。

## 小タスクの悪い例

- 「もっと良くする」
- 「可観測性を上げる」
- 「全体を改善する」

これらは goal ではなく方向性であり、フレーム goal としては広すぎる。

## 小タスクの良い例

- `self_check` の失敗理由を出力する
- `describe_agent` に 1 項目だけ追加して検証する
- `SyntaxError` の行だけを局所修復する
- validation stderr の原因箇所を特定する

## 親子の責務

- 親フレーム
  - 何を解くべきか決める
  - 子へ何を持たせるか決める
  - 子の結果を見て次の大きな判断をする
- 子フレーム
  - 局所 goal に集中する
  - 必要ならさらに下へ分解する
  - 結果か判断材料を持って親へ戻る

## 重要な一文

再帰フレームは「成功するまで下る仕組み」ではない。
再帰フレームは「今の階層で扱うには粗すぎる問題を、十分に狭くなるまで切る仕組み」である。

## runtime への落とし込み

- system は、planning を代行しすぎない
- system が出すのは「行動命令」ではなく「現在のフレーム状態」
- `warning` は補助情報としては使えるが、本質ではない

必要なのは次のような事実の開示である。

1. 現在のフレーム goal
2. 親がこのフレームに期待している返却物
3. いま局所 goal を満たしたか
4. 未達でも、親が次を判断するのに十分な情報がそろったか
5. この階層で続けると情報利得が低いか
6. さらに下へ行くべき局所問題が見つかったか

つまり runtime は、`return_ready` や `decompose_ready` のような状態を事実として見せる。
その状態を見て、

- `continue_here`
- `open_child_frame`
- `return_to_parent`
- `finish`

のどれを選ぶかは LLM が決める。

## hard rule を使ってよい範囲

強制してよいのは、planning ではなく safety / integrity に限る。

例:

- protected path は編集禁止
- patch の `old_text` は一致必須
- validation 成功前に promotion はしない

逆に、次を続けるか戻るか下るかは、原則として hard rule で固定しない。
ここは `P2` 自身が判断する領域である。
