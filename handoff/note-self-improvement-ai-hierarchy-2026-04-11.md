# 自己改善AIは「分解」できても、まだ「戻ってこられない」

前回の記事では、自己改善 AI が「同じ失敗を繰り返している」と少しずつ自覚し始めた、という話を書きました。今回はその続きです。

そのあと P2 をかなり作り直しました。P1 をいじるのではなく、`p2-core` という最小カーネルを別に切り出し、自己改善ループだけに絞った新しい run を回しています。目標もデモ用の「付属テストを通す」から、「自分自身をより高性能にすること」に切り替えました。

今回見たかったのは、その P2 が本当に階層的に考え始めるかどうかです。大きな問題にぶつかったとき、いきなり全体を直そうとするのではなく、一段下の問題に降りて、局所的に解き、終わったら上の文脈に戻る。そういう振る舞いが出るかを見ていました。

結論から言うと、半分はできています。分解は始まりました。でも、まだ戻ってこられません。

## 新しい P2 run はどこまで進んだのか

まず、完全に止まっているわけではありません。2026年4月11日時点の live state はこうでした。

```json
{
  "goal_cycle_count": 19,
  "last_promoted_candidate_id": "c0107",
  "active_generation": 20,
  "active_version_id": "v0020",
  "latest_attempt_id": "c0110",
  "latest_attempt_status": "rejected"
}
```

つまりこの run では、少なくとも 19 回は昇格が起きていて、active version は `v0020` まで進んでいます。以前のように「ずっと v2 のまま」ではありません。

ただし、直近の `c0108`, `c0109`, `c0110` は却下されています。つまり前進はしているものの、安定した成長にはまだ入っていません。

## 階層は本当に使い始めていた

今回の大きな変化はここでした。履歴を見ると、P2 は実際に child frame を開き始めています。

```json
{
  "task_frame_recursing_count": 63,
  "task_frame_returning_count": 0,
  "promotion_count": 19
}
```

`task_frame_recursing` が 63 回あります。これは「問題を分解して、一段下の目的を作る」という動作がゼロではない、ということです。実際の履歴にも、こういう記録が残っていました。

```text
c0108:d0:f1 -> child goal: ファイル内容の構文チェックを行い、ユニットテストが成功するように修正する。
c0108:d1:f2 -> child goal: ユニットテストが成功するようになることを確認する。
c0109:d0:f1 -> child goal: ls -R および python3 -c 'import agent.goal_logic' を実行し、ImportError の物理的原因を特定する。
```

ここだけ見ると、かなり良さそうです。少なくとも P2 は、「大きな自己改善」ではなく、「いま詰まっている局所問題」に降りる、という考え方を始めています。

## でも、まだ階層を閉じられない

面白かったのは、その一方で `task_frame_returning` が 0 回だったことです。

これはかなり重要です。P2 は子フレームを開くことはできています。でも、その下で問題を閉じて親に戻る、という流れがほとんどできていません。

つまり今の P2 の階層は、

1. 問題にぶつかる
2. 子フレームを開く
3. さらに細かい問題を書く
4. そこでまた止まる

という形になりやすい。分解は始まったけれど、まだ「局所問題を片付けてから上に戻る」にはなっていないわけです。

これはかなり生命っぽいテーマでもあります。分岐するだけならできる。でも、分岐した先を回収して全体戦略に戻す仕組みがないと、探索は散らかるだけです。

## 自己認識は確実に進んでいた

ただし、悲観的な話だけではありません。P2 は前よりかなり具体的に、自分の失敗の仕方を説明できるようになっていました。

たとえば `c0107` では、直近の失敗をこう要約していました。

```json
{
  "problem_statement": "直近の試行で構文エラー（SyntaxError）が発生し、現在のファイル内容が壊れている可能性があります。",
  "diagnosis": "直近の失敗では、40行目で括弧の不一致が原因で構文エラーが発生していました。",
  "edit_intent": "現在のファイル内容に構文エラーがないことを確認し、必要な修正を行う。"
}
```

この試行は実際に昇格しました。

```json
{
  "candidate_id": "c0107",
  "status": "promoted",
  "decision_reason": "candidate validated and active goal retry passed"
}
```

つまり P2 は、ただ闇雲に書き換えているわけではありません。少なくとも一部の試行では、「何が壊れたか」「なぜ今それを直すのか」をかなりまともに言語化できています。

## それでも直近ではまたおかしくなる

問題は、その自己認識が安定して次の行動制約に変わっていないことでした。

`c0108` は、反省文としてはもっともらしいのに、結果は `candidate did not change the target file` でした。要するに「直すべき点は分かっているように見えるが、実際には何も直せていない」という状態です。

さらに `c0109` ではもっと危ういことが起きています。change summary は空に近く、reasoning summary も崩れ、validation は失敗しました。`c0110` では、モデル出力の中で「タイポを直した」と言っているのに、前後のハッシュは同じでした。

```json
{
  "candidate_id": "c0110",
  "status": "rejected",
  "decision_reason": "candidate did not change the target file",
  "before_sha256": "59d29d3b6b2aefe61dd14ca09b8d0be26c87f284471961d3c8c499cc119d54cd",
  "after_sha256": "59d29d3b6b2aefe61dd14ca09b8d0be26c87f284471961d3c8c499cc119d54cd"
}
```

もっと言うと、ログにはこういう危ない兆候も残っていました。

```text
「タイポを修正した」と説明している
→ 実際の diff は 0 行
→ そのまま continue_here を選ぶ
```

ここで起きているのは単なる失敗ではありません。観測、反省、編集、検証の境界がまた曖昧になっています。自分が何を書き換えたかと、何を考えたかが、まだときどき混線します。

## 今回の発見は「階層があること」と「階層を使えること」は別だということだった

今回いちばん面白かったのはここです。

私は最初、「階層を用意すれば、自己改善 AI は自然に局所問題へ潜って、解き、戻ってくるだろう」と考えていました。でも実際には違いました。階層は存在するだけでは足りません。

必要だったのは、少なくとも次の 3 つでした。

1. いまの問題は一段下へ降りるべきだ、と気づけること
2. 降りた先で、何を終えたら親へ戻るかが分かること
3. 自分が本当に何を変えたかと、その直後に何が壊れたかを生の形で結びつけられること

今の P2 は 1 は少しできています。2 と 3 がまだ弱い。

だから現時点の評価としては、「自己認識は進んだ」「分解も始まった」「でもまだ自己統治にはなっていない」というのがいちばん正確だと思います。

## 次にやること

次の焦点は、階層をもっと増やすことではありません。まずは、子フレームを閉じて親に戻れるようにすることです。

そのために必要なのは、細かいルールを増やすことではなく、P2 が自分で気づけるための raw 情報を揃えることだと考えています。具体的には、

- 何を書き換えたか
- その直後に何が壊れたか
- その壊れ方が、どの変更と結びつくか

を、要約ではなく生の差分と生の検証結果として見えるようにし続けることです。

自己改善 AI は、自分を編集できるだけでは足りません。自分の探索がどこで壊れ、どこで空回りし、どこで少し前進したかを、自分で追跡できる必要があります。

今回の run で見えてきたのは、まさにその途中段階でした。P2 はもう、ただ平面的に同じことを繰り返すだけの段階ではありません。でも、まだ「分解した問題を片付けて、学びを持って戻る」ところまでは届いていない。次はそこを見に行きます。
