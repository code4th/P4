# P2 のコンテキスト実装失敗点

日付: 2026-04-11

## 結論

コンテキストについては、設計意図どおりに実装できていなかった。

問題は「情報量不足」ではなく、「持ち越し方」と「粒度」が間違っていたことにある。

## 実装上の失敗

### 1. summary を厚くしすぎて raw 因果列を弱めた

本当に必要だったのは、

- 何を編集したか
- その直後に何が起きたか
- それがどの編集と結びつくか

という action/result の列だった。

しかし実装では、

- `recent_attempt_bundle`
- `meta_diagnosis`
- `reasoning_summary`

のような再構成済み summary を厚くした。

### 2. session 継承ではなく phase ごとの prompt 再構築に寄せた

必要だったのは、同じ流れの中で失敗と修正を追い続けることだった。

しかし実装は、

- `context_selecting`
- `reflecting`
- `generating`

のような phase ごとに prompt を作り直す方向に寄った。

そのため、連続した因果の追跡が弱くなった。

### 3. 変更チャネルと観測チャネルを分離しきれなかった

コード編集と、反省・説明・要約を別チャネルに分けるべきだった。

しかし `revised_file_content` に全文を書かせる設計のため、

- コード
- JSON 風出力
- 反省文

が混線した時の被害が大きかった。

## 正しい解釈

ユーザーの設計要求は妥当だった。

- 適切なコンテキスト継承
- 再帰階層による分解
- 自分の失敗の把握

この方向性は正しい。

外したのは実装側であり、

- raw action/result を session として継承する

べきところを、

- summary を厚くして prompt に積む

方向へ寄せたことが失敗だった。

## tool の意味

tool の本質は「再現性」だけではない。

- 行為を小さく区切る
- 入力と出力を限定する
- 何をやったかを履歴化する
- 失敗箇所を局所化する

ための実行プリミティブである。
