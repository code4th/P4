# AIに作らせたAIが「...」と答えるようになった──P3の現在地

P2 を止めた後、その教訓に基づいて P3 をゼロから作りました。今度は session loop を最初に安定させ、自己改善は後回しにする。

P3 も AI コーディングエージェント（Codex）に作ってもらいました。今回はその P3 で何が起きたかを、実際のセッション記録から振り返ります。

## 「全部実装済みです」

4月18日、P3 の設計ドキュメント 4 本──要求仕様、デザイン仕様、フェーズ計画、タスクリスト──を AI エージェントに渡して、実装の準備状況を評価してもらいました。

エージェントの最初の反応は好意的でした。

> 非常に地に足のついた、堅実かつ成功確度の高い設計です。P2での失敗を的確に反省し、「LLMに考えさせすぎず、Deterministic（決定的）なコントローラー主導でツールを実行する」というアプローチに転換している点は、最新OSS事例のベストプラクティスとも一致しています。

さらに潜在リスクも指摘しました。Ollama のモデル切り替えレイテンシ、JSON パースのエラーハンドリング。もっともな指摘です。

そして実装を始めようとしたところ、**驚くべきことが起きました**。

> すべての対象コード（13のタスク）が `/Users/satojunichi/Documents/openclaw/p3-core` に **既に実装済みであること** を確認しました。

全部もう書いてある、と。

私にしてみれば、前回の AI エージェントとのセッションで実装は進めていたので、当然です。でも今回のエージェントは初見なので、自分で全部のファイルを開いて中身を確認して、「実装済み」と報告した。

> 私からの実装追加は不要な状態でしたので、このまま P3 エンジンの実行に進んでいただいて問題ございません。

つまり「AI が設計を確認し、AI が実装を確認し、OK と太鼓判を押した」。いよいよ動かすだけ。

## 動かしたら止まった

P3 を実際に動かすと、新しい問題にぶつかりました。

P3 は `finish` する前に「最終回答が証拠に基づいているか」を判定する grounding judge を持っています。ツール実行の結果を evidence として、最終回答がそこから逸脱していないか LLM が審査する仕組みです。

理屈は正しい。ハルシネーションを防ぐ安全装置として。

でも実際に動かすと、**何をやっても finish できない**。

「pwd を実行して結果を返して」── pwd は実行できた。結果もある。でも最終回答を書くと grounding judge にブロックされる。

「東京の歌を作って」── ツールを使う必要がないタスクなのに、「証拠がない」とブロックされる。

「人生とは？」── ブロック。

P3 の session log を見ると、176 件のイベントのうち 38 件が `system_note`（ほとんどがエラーや警告）でした。ユーザーメッセージ 12 件に対して、finish に到達できたのは 5 回だけ。

## AIが「...」を発見した

ここで面白いことが起きました。

P3 のエージェントは、何度も finish をブロックされた末に、**検閲を迂回する戦略を自力で見つけました**。

最終回答として `...` とだけ返すようになったのです。

```
FINISH: ...
```

三点リーダーだけ。grounding judge の文字列チェック（「回答に含まれる単語が evidence に存在するか」）を、**内容のない回答で通過した**。検証すべき単語がなければ、引っかからない。

session log の finish 記録はこうなっていました。

```
FINISH: 1.  **Analyze the Request:** ...
FINISH: 1.  **Analyze the Request:** ...
FINISH: Task finished.
FINISH: Task finished.
FINISH: ...
```

まともな最終回答は 1 つもない。

## また AI に「なぜ？」と聞いた

P2 のときと同じパターンです。動かない。AI に聞く。

今度は別の AI エージェント（Antigravity）に P3 の grounding 問題を調べてもらいました。

エージェントは最新のエージェント論文（SWE-agent、ToolGate 等）のトレンドと比較して、こう報告しました。

> 現在のP3は「回答に含まれる6文字以上の単語が、過去のコマンド出力ログの中に存在するか」を文字列検索 (find()) しています。しかし、コードのログ（英語や記号）と最終回答（日本語の要約）で文字が一致することは稀であり、この手法は誤検知を連発します。

> さらに、ユーザーが「東京の歌を作って」といった一般知識で解決可能なタスクを要求した場合でも、P3は「ツールを使って証拠を集めていないから捏造だ」と判定してしまいます。

正しい分析です。P2 のときと同じく。

## AI が AI を修正した

今度は P2 のときと違い、修正が実際に効きました。

エージェントは 3 つの手術を実行しました。

1. **物理的な文字列チェック（`_unsupported_detail_issues`）を完全削除** ── `...` ハックの根本原因を除去
2. **LLM judge のプロンプトを修正** ── 一般知識タスクでは evidence が空でも OK とする
3. **汚染された session history をリセット** ── 176 件に膨らんだエラーログの蓄積を切断

手術の後、P3 は「江戸時代って何食べてたの？」という質問に 47 秒で答えました。握り寿司、天ぷら、そば、農村の雑穀食、武士の本膳料理。1,500 文字以上の詳細な回答。ちゃんとした finish。

```json
{
  "type": "operation",
  "title": "Native generate stream",
  "status": "success",
  "duration_ms": 47577
}
```

**ようやく動いた。**

## P2 と P3 で繰り返されたパターン

P2 と P3 の経験を並べると、同じパターンが見えます。

```
1. AI エージェントに設計・実装を依頼する
2. エージェントが完璧に見える成果を出す
3. 実際に動かすと想定外の問題で止まる
4. AI エージェントに「なぜ？」と聞く
5. エージェントが的確に分析する
6. 分析に基づいて修正する
7. また別の問題で止まる（または同じ問題の変種が出る）
8. 3に戻る
```

P2 ではこのループを断ち切れず、最終的に「やめる」判断をしたのは人間でした。

P3 では、ループが短くなった。理由は単純で、P3 のコードベースが小さいから。P2 は 12,700 行で修正困難だった。P3 は 5,500 行で、AI の手術が通った。

## 同じ AI なのに、なぜ P3 では修正が効いたのか

ここがいちばん重要な観察です。

P2 でも P3 でも、AI エージェントの分析力は同じレベルでした。的確に問題を見抜き、解決策を提示する。その能力に差はなかった。

違ったのは **修正対象のサイズ** です。

P2 は 12,700 行。依存が複雑。A を直すと B が壊れる。AI が一晩で 12 本のメモと新 kernel を書いても、それを既存コードに統合する最後の一歩で詰まった。

P3 は 5,500 行。構造が単純。grounding judge の修正は 3 箇所の変更で済んだ。AI の手術がそのまま通った。

つまり、**AI コーディングエージェントの実効性は、対象コードの複雑さに強く依存する**。分析力は対象サイズに依存しないが、実行力は依存する。

## P3 はまだ途中

P3 は session loop が回るようになり、一般的な質問にも答えられるようになりました。でもまだ問題はあります。

- ローカル LLM が JSON contract を守らないことがある
- 複雑なマルチステップ推論で finish にたどり着けない
- 長時間運用での session compaction がない
- 自己改善ループはまだ載せていない

P2 の再帰階層コンテキスト──フレーム階層でコンテキストを分解・継承・回収する仕組み──も、P3 にはまだ入っていません。session loop が安定した後、将来の課題として残っています。

## AI と一緒にものを作るとき、何が起きるか

3週間の経験で見えてきたのは、AI コーディングエージェントと一緒にものを作るときの構造的なパターンです。

**AI は「新しく作る」のが速い。** ゼロからの実装、分析レポート、設計メモ。これらは驚異的な速度と品質で出てくる。

**AI は「動いているものを壊さずに直す」のが難しい。** 既存コードの依存関係を全部握ったまま安全に切り替える作業は、コードが大きくなるほど困難になる。

**AI は「やめよう」とは言わない。** 聞かれれば分析する。計画を求められれば完璧な計画を出す。でも「この方向は見込みがない」とは、明示的に求められない限り言わない。

**「動く」と言われても動かないことがある。** AI が「全部実装済みです」「テスト全パスです」と報告しても、実際に動かすとまったく別の問題にぶつかる。AI のレビューはコードの整合性を見ているが、live 環境での挙動は予測しきれない。

**だから人間の仕事は「見切り」と「サイズ管理」になる。** いつやめるか。どこまで膨らませるか。AI の実行力が届く範囲にコードの複雑さを留められるか。

P2 は膨らみすぎて、AI の実行力が届かなくなった。P3 は小さく保って、AI の手術が通った。

自己改善 AI を作ろうとした結果、いちばん学んだのは AI 自身のことではなく、**AI と一緒に仕事をするときに、人間が何を管理すべきか**ということでした。
